# Fancy Agent 项目结构说明

本文档介绍 **Fancy Agent** 项目的整体目录结构与各模块职责，便于新成员快速了解代码布局与扩展方式。

---

## 一、项目概览

Fancy Agent 是一个前后端分离的 **AI 智能体应用**，后端基于 **FastAPI**，前端基于 **React + Vite + TypeScript**，集成了 LangChain/LangGraph、MCP（Model Context Protocol）等能力，支持多 LLM、多 Agent、会话管理与文件上传。

```
fancy_agent/
├── backend/          # 后端服务（Python / FastAPI）
├── frontend/         # 前端应用（React / Vite / TypeScript）
├── README.md         # 项目说明与启动方式
└── .gitignore
```

---

## 二、后端结构（backend）

后端采用 **FastAPI**，使用 **SQLAlchemy** 做异步 ORM，**uv** 管理依赖，数据库为 **PostgreSQL**（asyncpg）。

### 2.1 顶层目录

| 路径 | 说明 |
|------|------|
| `app/` | 应用主代码目录 |
| `db_init/` | 数据库初始化 SQL（建表等） |
| `pyproject.toml` | 项目依赖与元信息（uv） |
| `uv.lock` | 依赖锁定文件 |
| `.env` | 环境变量（数据库、密钥等，不提交） |

### 2.2 app 目录结构

```
app/
├── main.py              # 应用入口：FastAPI 实例、路由挂载、中间件、生命周期
├── api/                 # HTTP 路由（REST API）
├── core/                # 核心配置与基础设施
├── deps/                # 依赖注入（DB、当前用户、Service/Mapper 等）
├── middleware/          # 请求/响应中间件
├── models/              # SQLAlchemy 数据模型（表映射）
├── mappers/             # 数据访问层（CRUD、ORM 封装）
├── schemas/             # Pydantic 请求/响应模型与 DTO
├── services/            # 业务逻辑层
└── utils/               # 通用工具与 LangChain 相关封装
```

### 2.3 各模块说明

#### api/ — 路由层

| 文件 | 职责 |
|------|------|
| `auth_router.py` | 登录、注册、Token 等认证相关接口 |
| `agent_router.py` | Agent 的 CRUD |
| `agent_mcp_router.py` | Agent 与 MCP 的关联管理 |
| `chat_router.py` | 对话/聊天接口（消息发送、流式等） |
| `llm_router.py` | LLM 模型配置的 CRUD |
| `mcp_router.py` | MCP 的 CRUD |
| `session_router.py` | 会话的 CRUD |
| `upload_router.py` | 文件上传 |

路由在 `main.py` 中统一 `include_router` 挂载，并配置 CORS、请求日志等中间件。

#### core/ — 核心配置与基础设施

| 文件 | 职责 |
|------|------|
| `config.py` | 从环境变量读取配置（数据库、JWT、日志、应用端口等） |
| `database.py` | 数据库连接与会话管理（异步） |
| `logging_config.py` | 日志初始化与格式配置 |
| `security.py` | 密码哈希、JWT 等安全相关 |

#### deps/ — 依赖注入

| 文件 | 职责 |
|------|------|
| `db.py` | 提供数据库 Session（get_db 等） |
| `user.py` | 获取当前登录用户（get_current_user） |
| `service.py` | 注入各 Service 实例 |
| `mapper.py` | 注入各 Mapper 实例 |

供路由层通过 `Depends()` 使用，实现解耦与可测试性。

#### models/ — 数据模型（ORM）

与数据库表一一对应，例如：

| 文件 | 对应概念 |
|------|----------|
| `user.py` | 用户 |
| `llm.py` | LLM 配置 |
| `mcp.py` | MCP 服务 |
| `tool.py` | MCP 下的工具 |
| `agent.py` | Agent |
| `agent_mcp.py` | Agent–MCP 关联 |
| `session.py` | 会话 |
| `chat_message.py` | 聊天消息 |
| `chat_file.py` / `chat_file_content.py` | 聊天相关文件与内容 |
| `timestamp_model.py` | 时间戳基类 |

#### mappers/ — 数据访问层

对 models 的 CRUD 封装，供 services 调用，例如：

- `user_mapper.py`、`llm_mapper.py`、`mcp_mapper.py`、`tool_mapper.py`
- `agent_mapper.py`、`agent_mcp_mapper.py`
- `session_mapper.py`、`chat_message_mapper.py`、`chat_file_mapper.py`、`chat_file_content_mapper.py`
- `base_mapper.py` 可放通用基类

#### schemas/ — 请求/响应模型

Pydantic 模型，用于 API 入参校验与出参序列化：

- `auth.py` — 登录/注册等
- `agent_schema.py`、`llm_schema.py`、`mcp_schema.py`、`tool_schema.py`
- `session_schema.py`、`chat_schema.py`、`chat_file_schema.py`
- `dto/langchian.py` — LangChain 相关 DTO

#### services/ — 业务逻辑层

| 文件 | 职责 |
|------|------|
| `user_service.py` | 用户相关业务 |
| `token_service.py` | Token 签发与校验 |
| `llm_service.py` | LLM 配置业务 |
| `mcp_service.py` | MCP 业务 |
| `agent_service.py` | Agent 业务 |
| `agent_mcp_service.py` | Agent–MCP 关联业务 |
| `session_service.py` | 会话业务 |
| `chat_service.py` | 对话流程编排（含 LangChain/LangGraph） |
| `chat_message_service.py` | 消息持久化与查询 |
| `file_upload_service.py` | 文件上传与存储 |
| `storage/base.py`、`storage/local.py` | 存储抽象与本地实现 |

#### middleware/ — 中间件

- `request_logging.py` — 请求/响应日志（可配置跳过路径、是否记录 body）

#### utils/ — 工具与 LangChain 封装

- `password_util.py`、`file_util.py`、`text_file.py` — 通用工具
- `langchain/agent_util.py` — Agent 构建与运行
- `langchain/message_processor.py` — 消息格式转换
- `langchain/middleware.py` — LangChain 相关中间逻辑
- `langchain/tool_util.py` — MCP 工具封装与调用

### 2.4 数据库初始化（db_init）

- `01table_create.sql`：创建 `users`、`llms`、`mcps`、`tools`、`agents`、`agents_mcps`、`sessions`、`chat_message` 等表及索引，与 `models/` 对应。

## 三、前端结构（frontend）

前端为 **React 18+** + **Vite 7** + **TypeScript**，使用 **Tailwind CSS**、**React Router**、**Axios**，并通过 OpenAPI 生成 API 客户端。

### 3.1 顶层目录

| 路径 | 说明 |
|------|------|
| `src/` | 前端源码 |
| `index.html` | 入口 HTML |
| `vite.config.ts` | Vite 配置 |
| `tsconfig.json` / `tsconfig.app.json` / `tsconfig.node.json` | TypeScript 配置 |
| `package.json` | 依赖与脚本（npm） |
| `.env.development` | 开发环境变量（如 API 基地址） |

### 3.2 src 目录结构

```
src/
├── main.tsx           # 应用挂载入口
├── App.tsx            # 根组件：路由、Layout、全局 Provider
├── index.css          # 全局样式（含 Tailwind）
├── api/               # 后端 API 客户端与类型
├── components/        # 可复用 UI 组件
├── context/           # React Context（全局状态）
├── hooks/             # 自定义 Hooks
├── pages/             # 页面级组件（与路由对应）
└── utils/             # 前端工具函数
```

### 3.3 各模块说明

#### api/ — 接口层

由 OpenAPI 生成 + 手写扩展，与后端 REST 一一对应：

- **core/**：请求封装、OpenAPI 配置、`ApiResult`、`ApiError`、`CancelablePromise` 等
- **models/**：请求/响应类型（如 `AgentOut`、`AgentCreate`、`SessionOut`、`ChatRequest`、`LLMOut`、`MCPOut` 等）
- **services/**：按资源划分的 API 方法，例如：
  - `AgentsService`、`AgentMcpsService`
  - `LlmModelsService`
  - `McpService`
  - `SessionsService`
  - `ChatService`（如流式对话等）
- `index.ts`：统一导出，供页面与 Context 使用

#### components/ — 通用组件

| 文件 | 职责 |
|------|------|
| `Layout.tsx` | 整体布局（侧边栏 + 主内容区） |
| `Sidebar.tsx` | 侧边导航（会话列表、Agent 选择等） |
| `Message.tsx` | 单条聊天消息展示（含 Markdown、代码高亮等） |
| `Modal.tsx` | 通用弹窗 |
| `AgentForm.tsx` | Agent 创建/编辑表单 |
| `LLMForm.tsx` | LLM 配置表单 |
| `MCPForm.tsx` | MCP 配置表单 |

#### context/ — 全局状态

- `AppContext.tsx`：提供全局的 agents、llmModels、mcpTools、sessions 以及当前选中的 agent/session，在应用启动时拉取列表数据，供侧边栏与聊天页使用。

#### hooks/ — 自定义 Hooks

- `useMessageHandler.ts`：处理发送消息、流式响应、历史消息加载等，封装聊天页与后端的交互逻辑。

#### pages/ — 页面

| 文件 | 路由 | 职责 |
|------|------|------|
| `ChatPage.tsx` | `/chat`、`/chat/:id` | 主聊天界面，按会话 id 加载历史 |
| `AgentsPage.tsx` | `/agents` | Agent 列表与创建/编辑 |
| `SettingsPage.tsx` | `/settings` | 设置页（LLM、MCP 等配置） |
| `MCPDetailPage.tsx` | `/mcp/:id` | 单个 MCP 详情与工具列表 |

#### utils/ — 工具

- `TokenManager.ts`：Token 的存取与附带在请求头
- `ApiClient.ts`：对 axios 或 OpenAPI 客户端的封装（若存在）

## 四、数据流与请求链路

1. **前端**：用户操作 → 页面/组件 → 调用 `api/services` 或 `hooks` → 通过 `utils` 带 Token 请求后端。
2. **后端**：请求进入 `api/*_router` → 使用 `deps` 注入的 Session、User、Service → Service 调用 Mapper 与 `utils/langchain` 完成业务与 AI 调用 → 返回 Pydantic schema 序列化结果。
3. **全局状态**：前端通过 `AppContext` 维护 agents、sessions、llm、mcp 等列表与当前选中项，聊天数据由 `useMessageHandler` 与 `ChatService` 管理。

---

## 五、如何基于现有结构扩展

- **新增后端接口**：在 `api/` 增加或修改 router，在 `schemas/` 增加请求/响应模型，在 `services/` 和 `mappers/` 中补全业务与数据访问，最后在 `main.py` 中挂载新 router。
- **新增前端页面**：在 `pages/` 新增页面组件，在 `App.tsx` 的 `Routes` 中增加 `Route`，如需新接口则在 `api/models` 与 `api/services` 中补充类型与方法（或重新生成 OpenAPI 客户端）。
- **新增数据表**：在 `db_init/` 增加建表 SQL，在 `models/` 增加对应模型，在 `mappers/` 和 `schemas/` 中增加访问与 DTO，再在相应 service/router 中使用。

## 六、快速启动（与 README 一致）

**后端：**

```bash
cd backend
uv sync
uv run uvicorn app.main:app
```

**前端：**

```bash
cd frontend
npm install
npm run dev
```

确保 PostgreSQL 已运行并执行过 `db_init/01table_create.sql`，且 `backend/.env` 中 `DATABASE_URL` 等配置正确。
